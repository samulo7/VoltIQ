# Repository Guidelines

## 项目结构与模块组织
当前仓库仅包含文档，所有文件位于仓库根目录：

- `AI_售电_产品设计文档.md`: 产品设计文档（主要规范）。
- `tech-stack.md`: 技术栈建议。
- 若后续新增文档，默认放在根目录；如引入 `docs/`，请统一迁移并更新本指南。

如果新增代码，请建立清晰的顶层划分，例如 `src/`、`tests/`、`docs/`，并在此处说明。

## 构建、测试与开发命令
目前未配置构建、测试或本地运行命令（仓库无代码与工具链）。若未来加入工具，请明确写出命令及用途，例如：

- `npm run dev`：本地开发。
- `pytest`：后端测试。

## 编码风格与命名规范
当前内容以 Markdown 为主，保持风格统一：

- 使用 `#`/`##` 标题与短段落。
- 结构化信息优先用表格。
- 控制单行长度，避免过长的表格行。
- 文件命名要描述清晰；中英混用时需在同一类文档内保持一致。

## 测试指南
当前无测试。若新增测试，请建立 `tests/` 目录并明确：

- 测试框架（如 `pytest`、`vitest`）。
- 命名规则（如 `test_*.py`、`*.spec.ts`）。
- 运行命令（具体可执行命令）。

## 提交与 PR 指南
当前无法读取 Git 历史，提交规范未知。若建立约定，请在此说明（如 Conventional Commits）。PR 需包含：

- 变更摘要。
- 关联问题/决策链接。
- 影响 UI/UX 或规格变更时的截图或文档更新。

## 安全与配置提示
本仓库为业务/产品文档，请避免提交敏感客户数据、凭据或真实电价数据。若引入配置文件，务必将密钥排除在版本控制之外，并在此列出必要环境变量。

## Agent 使用说明
编辑文档时请保持原有结构与语气，若修改需求请同步更新版本/日期字段，并确保内容范围与 `AI_售电_产品设计文档.md` 一致。

## AI 开发规则（强制）
以下规则面向 AI 开发者，必须严格遵守。

### Always（始终应用）
- 写任何代码前必须完整阅读 `memory-bank/architecture.md`（包含完整数据库结构）。  
- 写任何代码前必须完整阅读 `memory-bank/AI_售电_产品设计文档.md`。  
- 每完成一个重大功能或里程碑后，必须更新 `memory-bank/architecture.md`。  
- 必须坚持模块化与多文件拆分，禁止单体巨文件（monolith）。  

### 其他规则（推荐但非 Always）
- 后端采用 FastAPI 模块化组织（路由、服务、数据访问分层），避免业务逻辑堆叠在路由层。  
- 数据访问集中在仓储/DAO 层，禁止在控制器直接写复杂 SQL。  
- 异步任务统一使用 Celery + Redis，并为任务添加幂等性与重试策略说明。  
- 前端遵循 Ant Design Pro 约定，页面与组件分离，状态管理保持单一来源。  
- 依赖接入（如 Dify）必须封装为独立模块，禁止散落调用。
